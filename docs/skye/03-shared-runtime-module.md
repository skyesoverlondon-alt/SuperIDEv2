# SKYE-03 — Shared Runtime Extraction Plan

## Objective
Define a single browser-safe runtime path for `.skye` serialization, validation, encryption, and decryption.

## Required Reads
1. `docs/skye/02-contract-spec.md`
2. `src/App.tsx`
3. `src/lib`
4. `SkyeDocxPro/index.html`

## Target Files
- new shared utility under `src/lib/`
- `src/App.tsx`
- standalone surface consumption path for browser-only apps

## Edit Directives
- Define a shared module that owns:
  - `SkyeEncryptedBlock`
  - `SkyeSecureEnvelope`
  - `SkyePlainPayload`
  - `serializeSkyeEnvelope()`
  - `parseSkyeEnvelope()`
  - `validateSkyeEnvelope()`
  - `encryptSkyePayload()`
  - `decryptSkyePayload()`
- Decide how standalone HTML surfaces consume shared logic:
  - bundled shared script
  - synced utility artifact
  - other browser-safe distribution path
- Eliminate per-app crypto rule ownership.
- Keep the module browser-compatible and explicit about secure defaults.

## Validation
- One shared runtime path documented.
- Standalone loading strategy documented.
- No app should need to define its own contract constants after migration.

## Exit Criteria
- Shared runtime design is concrete enough to implement.
- Type and function surface is named and scoped.
- Source of truth for browser distribution is decided.

## Known Blockers
- Direct-public HTML surfaces may need a synced helper rather than import-based bundling.

## Runtime Design Decision
Primary source of truth will be a new TypeScript module at `src/lib/skye/skyeSecure.ts`.

Planned companion files:
- `src/lib/skye/index.ts` for shell-side exports
- `public/_shared/skye/skyeSecure.js` as the browser-consumable artifact for standalone HTML surfaces
- `scripts/sync-skye-runtime.js` to copy or build the browser artifact from the TypeScript source before source-root and direct-public migrations

## Exact Shared API Surface
The shared module must export:

- constants
  - `SKYE_BINARY_MARKER = "SKYESEC1"`
  - `SKYE_SECURE_FORMAT = "skye-secure-v1"`
  - `SKYE_DEFAULT_ITERATIONS = 120000`
- types
  - `SkyeEncryptedBlock`
  - `SkyeSecureEnvelope`
  - `SkyePlainPayload`
  - `SkyeLegacyEnvelopeAdapter` for migration-only readers
- helpers
  - `bytesToBase64()`
  - `base64ToBytes()`
  - `deriveSkyeKey()`
  - `encryptSkyePayload()`
  - `decryptSkyePayload()`
  - `serializeSkyeEnvelope()`
  - `parseSkyeEnvelope()`
  - `validateSkyeEnvelope()`
  - `readSkyeEnvelopeFromBlob()`
  - `normalizeLegacySkyeEnvelope()`

## Distribution Strategy
1. The React shell imports `src/lib/skye/skyeSecure.ts` directly.
2. Standalone source-root and direct-public surfaces consume `public/_shared/skye/skyeSecure.js` through a plain browser script include.
3. No standalone surface should retain app-specific crypto constants after migration.
4. `package.json` already has source-surface sync scripts; the runtime sync step should be added as a future predev / prebuild dependency after the shared module lands.

## Stage Deliverable
This stage is complete when the shared module location, public browser artifact path, and migration consumption model are fixed. Code implementation may begin immediately after this document is accepted.

## Stage Outcome
Complete. The shared TypeScript runtime now exists at `src/lib/skye/skyeSecure.ts`, the shell-facing barrel exists at `src/lib/skye/index.ts`, the standalone browser artifact is generated to `public/_shared/skye/skyeSecure.js`, and `scripts/sync-skye-runtime.js` keeps the browser artifact aligned with the TypeScript source.
