SkyDex Tutorial

How to actually operate SkyDex 4.6

SkyDex is a workspace IDE, release lane, and bounded agent surface. You can drop in a full repo as a folder or ZIP, edit the codebase, and let the agent autonomously iterate and save the workspace, but it is still sandboxed by auth, SKNore policy, the operation budget, and the max pass count.

What It Really Does

Real autonomy, inside a hard boundary

Autonomous execute mode can inspect the workspace snapshot, stage full-file operations, iterate across a virtual copy of the repo, then auto-apply and save the result. That means real codebase work after a full repo import, not just toy prompts. It still does not become an unrestricted desktop agent, bypass external auth, provision secrets, or run arbitrary shell commands from inside this surface.

Operating Flow

Six-step workflow users should follow

1. Load or import a workspace
Open the saved workspace, choose a SkyeDrive source, or drag in a folder or ZIP. Desktop drop imports a real repo snapshot into the live workspace, not a throwaway buffer.
2. Inspect the baseline
Check the snapshot diff, workspace intelligence, and active file. The agent works best when the workspace is current and the active path is meaningful.
3. Pick a delivery mode
Use controlled plan mode for review-heavy work. Use autonomous execute when you want the agent to make bounded multi-pass changes and save them automatically.
4. Add operator memory
Write local rules like “preserve route names,” “prefer focused edits,” or “do not touch auth.” This is the fastest way to steer the agent without repeating yourself.
5. Review staged operations
Read the reply, inspect the operation list, and apply manually when you are in controlled mode. In autonomous mode, the app will auto-apply and save if operations were returned.
6. Ship from the same state
Once the snapshot is correct, save and sync to SkyeDrive, then push to GitHub or deploy to Netlify. All release actions use the same saved workspace state.
When To Use Which Mode

Decision guide for operators

Controlled project
Use this for production edits, client review, large refactors, contract work, and anything where a human must approve the exact changes first.
Autonomous execution
Use this for repetitive implementation, documentation updates, scaffolding, cleanup, and bounded feature work where auto-apply is acceptable.
Plan preview first
Use plan mode whenever the request is ambiguous, wide in scope, or likely to touch multiple files and release-critical paths.
If the prompt is vague, ask for a plan first instead of letting the agent guess at scope.
If the workspace contains protected or sensitive files, keep operator memory explicit about what must not be touched.
If the run returns zero operations, treat that as signal: either the prompt is under-specified or the requested change is outside the workspace boundary.
If the agent is set to autonomous execute, remember that it can auto-apply and save; operators should not use that mode casually on release-critical work.
Recommendation: default new users to controlled plan mode until they understand the release path, then let them switch to autonomous execute for bounded implementation work.
